# CI Build Optimization Plan ## Scope This document governs Rust backend CI optimization for: - lint/check/test gates - Rust coverage gates - release-path execution on `main` and `release/*` branches The goal is to reduce redundant compilation in repeated release runs while keeping merge-gate semantics unchanged. ## Performance Criteria 1. **Feature-stable cache reuse** - release and non-release lanes must restore caches for the same compiler/toolchain and feature set - release lanes reuse a single compiled frontier whenever possible 2. **Single release prebuild boundary** - each release run executes exactly one `cargo test --bins --lib --tests --no-run` seed step - no duplicate full compilation is required for coverage and gate tests 3. **Parallel release wall-clock** - release build seed may run in parallel with quality gates (`cargo clippy`, `cargo check`, unit tests) 4. **Non-regression for correctness** - test selection, coverage artifact generation, and merge-gate expectations remain unchanged - behavior of the existing `CI gate contract` step is preserved 5. **Documented evidence and lock-in** - workflow changes are mirrored in this plan - CI contract tests assert the seed lane and cache behavior 6. **Release-only cache persistence** - `Swatinem/rust-cache` writes release artifacts only on `main` and `release/*` - non-release lanes reuse cache but do not rewrite release-state snapshots 7. **Release-floor de-duplication** - release jobs should consume the seeded frontier without extra full-target validation gates. - redundant post-seed compile steps are removed unless they change observable quality signal. ## Phases ### Phase 1 (Implemented) - Add `rust-build-seed` job with mode switch for release vs delta lanes. - Run a single full compile-frontier step: - `cargo test --locked --manifest-path src-tauri/Cargo.toml --bins --lib --tests --no-run` ### Phase 1b (Implemented) - Enable seed job cache persistence only on release branches. - Keep non-release lanes using restore-only cache behavior. ### Phase 2 (Implemented) - Add CI contract tests that assert: - seed job existence and branch gating - one `--bins --lib --tests --no-run` release seed command - `test` and `rust-coverage` lane dependency on seed cache frontier ### Phase 2b (Implemented) - Maintain overlap by running seed and quality gates without hard sequencing. - Coverage and quality are independent of each other once release seeding exists; each lane starts from the seeded frontier in the release branch. ### Phase 3 (Implemented) - Split build seeding into release and delta lanes: - release mode (`main` and `release/*`) runs one full frontier: - `cargo test --locked --manifest-path src-tauri/Cargo.toml --features analytics --bins --lib --tests --no-run` - non-release mode runs a delta frontier: - `cargo test --locked --manifest-path src-tauri/Cargo.toml --lib --tests --no-run` - Run non-release quality gates in delta scope to avoid unnecessary full target work: - `cargo clippy --lib -- -D warnings` - `cargo test --lib` - skip `cargo check --bins` on non-release lanes - Run coverage only on release lanes: - coverage remains available for release verification and avoids forcing full compile on each PR lane. ### Phase 3b (Implemented) - Add explicit per-job duration telemetry in logs so release- and delta-mode wall-clock deltas are tracked in CI run summaries. - Add targeted assertions for release-vs-delta command shape when the workflow is edited. - Extend plan with release-build floor metric that proves non-release runs do not execute the `--bins --lib --tests --no-run` frontier except in release mode. ### Phase 4 (Implemented) - Run release coverage without waiting on quality test completion so it overlaps with the `test` job on gated release pipelines. - Reuse seeded artifacts in coverage by passing `cargo llvm-cov --no-clean`, avoiding extra full workspace tear-down/rebuild between quality and coverage lanes. ### Phase 5 (Implemented) - Make the `test` job consume the seeded frontier by adding `needs: [rust-build-seed]` so quality gates reuse a fully populated seed cache on release lanes instead of duplicating full compile work. - Restrict `Swatinem/rust-cache` writeback in `rust-build-seed` to release branches only (`main`, `release/*`) while keeping restore behavior intact for all lanes. ### Phase 6 (Implemented) - Introduce analytics feature split in Rust manifest: - mark `duckdb` optional and bind it behind an `analytics` feature - keep release default features including `analytics` for full-feature workflows - require `--no-default-features` in non-release lanes to avoid DuckDB compile work - Update CI lane commands for split execution: - release seed: `cargo test --features analytics --locked --manifest-path src-tauri/Cargo.toml --bins --lib --tests --no-run` - delta seed: `cargo test --locked --no-default-features --lib --tests --no-run` - Gate DuckDB-heavy test modules under `#[cfg(feature = "analytics")]`: `baseline_delta`, `release_baseline_handler`, `release_baseline_storage`, `release_baseline`, `admin_service`, `admin_ingestion_guard`, `analytics_snapshot`, `ingestion_transport`, `storage_duallayer`, `tauri_command`. ### Phase 7 (Implemented) - Scope non-release CI gate-contract execution to delta feature floor: - on release lanes, run contract tests with default features (analytics) - on delta lanes, run contract tests with --no-default-features - Add lane-specific CI contract coverage for the CI gate step command shape so delta lanes cannot accidentally pull DuckDB in non-release workflows. ### Phase 7b (Implemented) - Remove redundant release `cargo check --bins` quality gate after seed because release lint already performs the release-seed compile floor, and the test gate already exercises library/test artifacts. - Keep release quality behavior on: - `cargo clippy --features analytics -- -D warnings` - `cargo test --lib --tests` - Update release acceptance evidence to require no duplicate post-seed bin-only gate. ## Acceptance Criteria - Release gate and coverage jobs no longer trigger avoidable repeated full compiles after seeding. - Non-release lanes use release-mode delta seeds and delta-quality scopes and do not invoke release-only frontier compilation. - Release wall-clock improves where compilers overlap with seed and quality work. - Release wall-clock improves further by running coverage in seed-overlapped release path (`rust-coverage` depends on `rust-build-seed` and not on `test`). - Cache writes remain release-sensitive and can still be reused across non-release runs for delta builds. - Coverage remains available on release lanes without adding non-release full rebuild burden. - CI contract tests cover: - release-only vs delta seed behavior - release-only coverage gating - lane-specific command shapes for clippy and tests - lane-specific command shape for CI gate contract execution - cache key persistence in seed and gate jobs - release vs delta feature selection (`--features analytics` vs `--no-default-features`) - removal of duplicate release bin-only check
