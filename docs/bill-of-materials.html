`test` aggregate job in `.github/workflows/ci.yml` is the merge gate for protected-branch checks and depends on parse/scope/lint/test/coverage jobs. ## 5) Release-critical validation entry points - Rust formatting: `cargo fmt --manifest-path src-tauri/Cargo.toml --all -- --check` - Rust linting: `cargo clippy --manifest-path src-tauri/Cargo.toml --all-targets -- -D warnings` - Rust tests: `cargo test --manifest-path src-tauri/Cargo.toml` - Rust coverage: `cargo llvm-cov --locked --manifest-path src-tauri/Cargo.toml --lcov --output-path target/coverage/lcov.info` - UI typecheck: `pnpm exec tsc -b` from `ui/` - UI unit tests: `pnpm exec vitest run` from `ui/` - Headless browser tests: `pnpm exec playwright test` from `ui/` ### Coverage artifacts - Backend Rust LCOV artifact: `target/coverage/lcov.info` - CI artifact name: `rust-coverage-lcov` - CI job name: `rust-coverage` - Current policy: informational baseline reporting only; no `--fail-under-*` threshold is enforced until a measured baseline is adopted. ## 6) Publish-gate summary Before any release reaches `origin/main`, the repository should satisfy: 1. Security controls are present and active. 2. Documentation points at the current release gate. 3. Rust and UI validation are green on the release branch. 4. Backend Rust coverage generates `target/coverage/lcov.info` and publishes the `rust-coverage-lcov` artifact. 5. GTK-free host migration artifacts stay in sync with the feature list, bead tracker, and test plan. 6. GitHub Actions checks are green on the PR or release branch. 7. The final publish uses a protected-branch-safe squash or PR merge. 8. `origin/main` remains the source of truth; local and remote must be reconciled through merge checkpoints. This BOM is the release inventory; the checklist in `docs/publish-readiness-checklist.html` is the executable gate.
